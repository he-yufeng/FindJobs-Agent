Agent 总体：采用「检索增强生成 + 工具编排 + 会话缓存」架构，对外统一接口 `answer(user_query) -> answer, trace`。

1. **检索模块（Retriever）**

   * 职责：从向量库/全文库中找与问题最相关的文档片段。
   * 接口示例：`retrieve(query, k=5) -> List[DocChunk]`，输入用户问题或经 LLM 重写的 query，输出带 `text, source_id, score` 的片段。

2. **LLM 推理模块（Reasoner）**

   * 职责：解析用户意图、规划是否检索/调工具、综合文档与历史记忆生成回答。
   * 接口示例：

     * `plan(user_query, memory, retrieved_docs) -> Plan(tool_calls, need_retrieve)`
     * `generate_answer(user_query, docs, tool_results, memory) -> answer, new_memory_items`。

3. **工具调用模块（Tool Manager）**

   * 职责：统一封装如计算器、数据库/搜索 API 等外部工具，并按 LLM 规划顺序执行。
   * 接口示例：`call_tool(name, args) -> result`，对 LLM 暴露为结构化函数调用模式，例如 JSON：`{"tool": "search", "args": {"q": "..."} }`。

4. **短期记忆模块（Short-term Memory / Buffer）**

   * 职责：保存最近若干轮对话摘要与关键中间结论，供多轮问答时引用，避免重复检索与推理。
   * 接口示例：

     * `read_memory(session_id) -> MemoryState(summaries, key_facts)`
     * `write_memory(session_id, new_items) -> None`。

调用流程：`answer()` 先读短期记忆 → 由 LLM 规划是否 `retrieve()` 与 `call_tool()` → 收集 doc 片段与工具结果 → LLM 生成最终回答并更新 `write_memory()`。

# ✅ 防误触下单（Anti-Mistap Ordering）如何实现？

### **① 双确认机制（Two-Step Confirmation）**

- 用户点击“下单”不会立即执行，而是弹出二次确认页：
  - 大字号标题：“确认购买？”
  - 单一主按钮“确定下单”
  - 次按钮“返回修改”
- 防止用户手滑或屏幕误触导致直接支付。

### **② 行为门槛设计（Intent Verification）**

- 要求用户执行一个**明确意图动作**：
  - 左右滑动确认（如“滑动下单”）
  - 持续按住1秒完成下单（类似长按付款）
- 能显著减少无意点击。

### **③ 关键按钮尺寸与安全距离**

- “下单/支付”按钮必须≥48dp，四周需留出安全空区，避免被其他元素挤贴。
- 禁止将下单按钮放在浮层广告或弹窗附近。

### **④ 重要操作不可被遮挡**

- 避免页面跳动和元素因网络延迟重新布局导致误触（Layout Shift）。
